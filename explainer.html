Trading / Brokerage POCs

Six single-file Java 21 engines — the hard algorithmic core of a brokerage, de-risked before framework work begins.

What is this?

Each POC attacks the one thing that is genuinely hard in a brokerage subsystem — the part that is correct 99% of the time and catastrophically wrong in the 1% edge case, invisible until real money moves. Click a card to jump to its walkthrough.

Cross-cutting decisions

Opinions shared by all six, worth carrying into daccount.

Money in long microsInteger VND × 10⁶. No double drift, no BigDecimal allocation in hot paths. The only double is the margin ratio comparison.
Sealed interfaces + exhaustive switchA missing case is a compile error, not a production surprise. Used for events, decisions, corporate actions.
Single-writer per aggregateNo locks on the book/position. Deterministic. Same pattern as LMAX Disruptor / Akka.
Append-only event & audit logsNothing mutated in place. Enables back-testing, reconciliation, regulatory replay — and doubles as the recovery log.
Snapshots wrap irreversible opsCorporate actions keep before/after snapshots — the only way to investigate after the fact.

Order Matching OrderMatchingPoc.java

Match incoming orders against a resting book with correct price-time priority and partial fills — deterministically.

The hard problem

Correct price-time priority + partial fills, while a stop can convert to a market order mid-match and re-enter the same book. Wrong ordering = you fill the wrong customer first.

What we protect

  • Best price first; within a price, first-arrived first
  • Resting order's price wins (aggressor improvement)
  • Atomic: both sides + event log in one step
  • Determinism for replay & test

Data structure

bids: TreeMap<price↓, ArrayDeque<Order>>   // best bid first, FIFO within level
asks: TreeMap<price↑, ArrayDeque<Order>>   // best ask first, FIFO within level
stopBook: List<Order>                      // checked after every trade
incoming order→ cross() vs opposite book→ rest the remainder→ checkStops()

Live order book — send an aggressive order

BIDS
Watch the aggressive buy walk the ask side FIFO, take price improvement, and leave a partial fill resting.

Tech debt

  • Market remainder = IOC only (no FOK/GTC/post-only)
  • checkStops() is O(n) per trade — prod wants POC 13's TreeMap index
  • Single symbol/thread; no self-trade prevention, fees, or tick/lot validation

Real-time P&L RealtimePnlPoc.java

Value a live portfolio under tens of thousands of ticks/sec without recomputing everything.

The hard problem

Naïve engine recomputes the whole portfolio on every tick — O(positions) — and pushes it all to the client. Recompute only what ticked; push only what changed.

What we protect

  • FIFO lots for realized P&L
  • O(1) unrealized via incremental weighted-avg
  • Exactly one writer per position — no lost ticks
  • Client sees deltas, not snapshots

Two costing methods, one position

Realized (on sell)FIFO over ArrayDeque<Lot> — pop oldest lots; you must know which lot you sold.
Unrealized (on tick)O(1) = qty × (last − avgCost) using the incrementally maintained weighted average — no scan of lots per tick.
30k ticks→ single mailbox (ArrayBlockingQueue)→ virtual-thread pump→ route to the 1 position that ticked→ delta ≠ 0 ? push 
// only publish when the delta is non-zero — saves bandwidth on tiny ticks
if (next != prev) { sink.accept(pos); deltasPublished.incrementAndGet(); }

Tech debt

  • Incremental avg accumulates rounding — prod reconciles vs periodic full recompute
  • Mailbox drop-on-full loses ticks silently — needs seq numbers + gap detection
  • No corporate-action hook — a split mid-session corrupts basis until POC 16 wired in

Stop-Loss Engine StopLossEnginePoc.java

Fire triggers on a price cross over an interval — not on equality with the latest tick.

The hard problem: GAPS

Price can jump straight through a trigger between two ticks (overnight gap, lunch reopen). Fire on price == trigger and you silently miss the gap — the exact case the stop exists for.

What we protect

  • Downside protection works across gaps
  • Trailing stops ratchet, never give ground
  • OCO: one leg fires → the other is cancelled once

The core trick — range query, not equality

// Instead of "is the trigger == the new tick?", ask the TreeMap for EVERY
// trigger whose price lies in the band the price just moved through:
view = fallsTo.subMap(now, true, last, false);   // (now .. last]  — O(log n + k)

Gap-down demo — a stop at 120,000

stop 120,000
Price at 122,000. A naïve engine only fires if a tick lands exactly at 120,000.

Trailing & OCO

  • Trailing: on a favorable tick advance extremePrice, recompute extreme − trail; if higher, move the key in the TreeMap. Unfavorable ticks leave it put — the ratchet.
  • OCO: each leg holds the other's id; firing one deactivates & removes the other.

Tech debt

  • Restart loses resting triggers — must reload store and re-establish lastPrice before arming, or cold start mis-fires
  • Fires don't route to execution — that's POC 11
  • No halt/auction-reopen semantics

Margin Engine MarginEnginePoc.java

Pre-trade affordability + intraday revaluation + the full margin-call lifecycle.

The hard problem

The pre-trade check must decide before execution whether the account affords initial margin and won't immediately breach maintenance margin on the resulting mix — then the tick loop must run call → grace → cure/liquidate.

What we protect

  • Broker capital: risk never exceeds equity
  • Fair liquidation: weakest first, only as much as needed
  • Deterministic money math
equity = cash + Σ(qty × last) − marginLoan
ratio  = equity / positionValue          // margin call when ratio < weightedMM
tick→ reprice positions→ evaluateMargin→ NONE→ ISSUED→ CURED or LIQUIDATED
Pre-trade catchProjects the maintenance ratio including the prospective lot (weightedMMProjected) — rejects the "affordable now, margin-called instantly" trap.
Lifecycle guardenum CallState state machine makes double-issue / liquidate-a-cured-account impossible.
Liquidate weakestSort by loss %, sell worst-first, re-check ratio after each sale, stop at MM × 1.10 buffer.

Tech debt

  • Ratio math uses double (the one deliberate one) — prod needs a tolerance band to stop flapping at ratio ≈ MM
  • Liquidation is one-shot at last price — no child-order schedule/slippage
  • No cross-margin/netting; no derivatives Greeks

Settlement (T+2) SettlementPoc.java

Cash lives in two states at once: settled (withdrawable) and pending (reusable, not withdrawable).

The hard problem

Get the two-balance accounting wrong and you either block legitimate buys or — worse — let a customer withdraw money the depository hasn't delivered.

What we protect

  • Hard wall: only settled cash can leave
  • Rolling reuse: pending proceeds fund buys same-day
  • Append-only audit of every booking/settle/fail

The two formulas are the design

availableForBuy      = settled + pendingCredits − pendingDebits   // buys may spend pending
availableForWithdraw = settled              − pendingDebits        // withdrawals may NOT
T+0 sell booked→ CREDIT into T+2 bucket (pending)→ same-day buy reuses it→ T+2 day-tick sweep→ net into settled

A failed delivery marks the bucket FAILED; the sweep skips it and settled cash is never credited. The Status field is exactly the idempotency guard the distributed version needs (see Scaling).

Tech debt

  • Naïve weekend skip — needs a real holiday calendar
  • Instant.now() makes replay non-deterministic — inject a clock
  • No FX, partial settlement, or counterparty netting; failed-buy unwind undefined

Corporate Actions CorporateActionsPoc.java

Splits, dividends, mergers — applied at EOD with basis adjustment and a pre/post snapshot trail.

The hard problem

These rewrite positions + cost basis + open orders at once. Silent bug factories: a wrong split ratio makes all downstream P&L fictional and is unrecoverable after the fact without a snapshot.

What we protect

  • Basis integrity: 2:1 split halves basis with share count
  • Open orders adjust (₫160k limit → ₫80k)
  • Immutable before/after snapshot per action
announce → queue by effective date→ runEod(today)→ snapshot BEFORE→ mutate (sealed switch)→ snapshot AFTER→ AdjustmentEvent logged
2:1 Splitqty × 2, cost ÷ 2 — total value invariant. Open orders scaled the same way.
Cash dividendcash += qty × perShare (POC keeps it gross; WHT seam marked).
Stock dividendSame total cost spread over more shares.
MergerBasis (minus cash portion) carries into acquirer shares, combined by weighted avg; source orders cancelled.

Exhaustiveness: because CorporateAction is sealed, adding a new type (e.g. rights issue) is a compile error until every switch handles it.

Tech debt

  • Withholding tax skipped (marked in-code)
  • Integer division leaves basis residue — needs fractional/cash-in-lieu handling
  • Snapshots are full in-heap deep copies — prod externalises to an event store
  • No reversal flow for amended action terms

Scaling across pods / VMs

Every POC is single-JVM, single-writer. That's what makes them correct. The moment there are two replicas, that guarantee breaks — and you get wrong money numbers, not just lost performance.

What breaks with naïve pod scaling

Round-robin a keyed workload across N pods and: two P&L pumps mutate "the" VNM position and diverge; a margin check races a revaluation on another pod; two pods both run the settlement sweep and double-credit a bucket.

The fix: keep single-writer, but make it single-writer per aggregate across the whole cluster, not per process.

A · Partition by key hot engines

Kafka topic keyed by aggregate → all messages for a key hit one partition → one consumer = the single writer. Scale by partition count, not pod count.

K8s: StatefulSet for stable identity, long terminationGracePeriodSeconds, PodDisruptionBudget. Rebuild in-memory state on handoff before processing.

B · Leader / run-once batch jobs

Settlement sweep & corp-action EOD must run exactly once. Use K8s CronJob + concurrencyPolicy: Forbid, or a Redis SET NX lock. Idempotency still mandatory — guard with a persisted compare-and-set status.

C · Optimistic concurrency no sticky routing

DB is the source of truth; every write is UPDATE … WHERE id=? AND version=?. In-memory state is a cache. Good for the API-path margin check.

State recovery — the part you can't skip

Under partitioning a pod loses its partition at any time. The new owner must rebuild the exact in-heap state:

load latest snapshot→ replay events since snapshot offset→ then start consuming live

This is why "append-only everything" matters beyond audit — the audit log is also the recovery log.

Not claiming

These POCs are not cluster-ready as written — this is the design for making them so. No 2PC/distributed transactions: the pattern is partition + idempotent event replay, which the append-only design was chosen to enable. Full detail in CONSISTENCY.md.
